# 继续执行格式 (Continuation Format)
1: 
2: 
3: 完成命令或工作流后，展示下一步操作的标准格式。
4: 
5: ## 核心结构
6: 
7: ```
8: ---
9: 
10: ## ▶ 下一步
11: 
12: **{identifier}: {name}** —— {one-line description}
13: 
14: `{command to copy-paste}`
15: 
16: <sub>先执行 `/clear` → 获取全新的上下文窗口</sub>
17: 
18: ---
19: 
20: **还可进行：**
21: - `{alternative option 1}` —— description
22: - `{alternative option 2}` —— description
23: 
24: ---
25: ```
26: 
27: ## 格式规则
28: 
29: 1. **始终显示其内容** —— 名称 + 描述，绝不仅是命令路径
30: 2. **从源获取上下文** —— 阶段从 ROADMAP.md 获取，方案从 PLAN.md 的 `<objective>` 获取
31: 3. **行内代码形式的命令** —— 使用反引号，方便复制粘贴，渲染为可点击链接
32: 4. **`/clear` 说明** —— 始终包含，简明扼要地解释原因
33: 5. **使用“还可进行”而非“其他选项”** —— 听起来更像应用
34: 6. **视觉分隔符** —— 上下各使用 `---` 使其突出
35:

37: 
38: ### 执行下一个方案
39:

209: ## 坏做法
210: 
211: ### ❌ 不要：仅提供命令（无上下文）
212: 
213: ```
214: ## 继续操作
215: 
216: 运行 `/clear`，然后粘贴：
217: /gsd:execute-phase 2
218: ```
219: 
220: 用户不知道 02-03 是关于什么的。
221: 
222: ### ❌ 不要：缺少 /clear 说明
223: 
224: ```
225: `/gsd:plan-phase 3`
226: 
227: 先运行 /clear。
228: ```
229: 
230: 没解释原因。用户可能会跳过它。
231: 
232: ### ❌ 不要：使用“其他选项”措辞
233: 
234: ```
235: 其他选项：
236: - 查看路线图
237: ```
238: 
239: 听起来像是事后才想到的。应改为使用“还可进行：”。
240: 
241: ### ❌ 不要：在命令中使用围栏代码块
242: 
243: ```
244: ```
245: /gsd:plan-phase 3
246: ```
247: ```
248: 
249: 在模板内使用围栏代码块会导致嵌套歧义。请使用行内反引号。
