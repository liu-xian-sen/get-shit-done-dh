<purpose>
在规划之前，揭示 Claude 对某个阶段的假设，让用户能尽早纠正误解。
与 discuss-phase 的关键区别：这是对 Claude 所想内容的“分析”，而不是对用户所知内容的“摄取”。不产生文件输出——纯粹通过对话来引导讨论。
</purpose>

<process>

<step name="validate_phase" priority="first">
阶段编号：$ARGUMENTS（必填）

**如果缺少参数：**

```
错误：需要阶段编号。

用法：/gsd:list-phase-assumptions [phase-number]
示例：/gsd:list-phase-assumptions 3
```

退出工作流。

**如果提供了参数：**
验证路线图中是否存在该阶段：

```bash
cat .planning/ROADMAP.md | grep -i "Phase ${PHASE}"
```

**如果未找到阶段：**

```
错误：路线图中未找到阶段 ${PHASE}。

可用阶段：
[列出路线图中的阶段]
```

退出工作流。

**如果找到阶段：**
从路线图中解析阶段详情：

- 阶段编号
- 阶段名称
- 阶段描述/目标
- 提及的任何范围细节

继续执行 analyze_phase。
</step>

<step name="analyze_phase">
根据路线图描述和项目上下文，识别五个领域的假设：

**1. 技术方案：**
Claude 会使用哪些库、框架、模式或工具？
- "我会使用 X 库，因为……"
- "我会遵循 Y 模式，因为……"
- "我会将其结构化为 Z，因为……"

**2. 实施顺序：**
Claude 会先构建什么，然后是什么，最后是什么？
- "我会从 X 开始，因为它是基础"
- "然后是 Y，因为它依赖于 X"
- "最后是 Z，因为……"

**3. 范围边界：**
在 Claude 的理解中，哪些包含在内，哪些排除在外？
- "本阶段包括：A, B, C"
- "本阶段不包括：D, E, F"
- "边界模糊点：G 怎么做都可以"

**4. 风险区域：**
Claude 预计哪里会有复杂性或挑战？
- "棘手的部分是 X，因为……"
- "潜在问题：Y, Z"
- "我会留意……"

**5. 依赖关系：**
Claude 假设哪些已经存在或需要就位？
- "这假设了前序阶段的 X"
- "外部依赖：Y, Z"
- "这将被……使用"

诚实面对不确定性。用置信度标记假设：
- "相当有信心：……"（从路线图中可以明确得知）
- "假设：……"（合理的推论）
- "不明确：……"（可能有多种方式）
</step>

<step name="present_assumptions">
以清晰、易读的格式展示假设：

```
## 我对阶段 ${PHASE} 的假设：${PHASE_NAME}

### 技术方案
[列出关于如何实施的假设]

### 实施顺序
[列出关于顺序的假设]

### 范围边界
**在范围内：** [包括的内容]
**在范围外：** [排除的内容]
**模糊点：** [怎么做都可以的内容]

### 风险区域
[列出预期的挑战]

### 依赖关系
**来自前序阶段：** [需要的内容]
**外部：** [第三方需求]
**输出至：** [未来阶段需要本阶段提供的内容]

---

**你觉得怎么样？**

这些假设准确吗？请告诉我：
- 我哪些地方说对了
- 我哪些地方说错了
- 我遗漏了什么
```

等待用户响应。
</step>

<step name="gather_feedback">
**如果用户提供更正：**

确认更正：

```
关键更正：
- [更正 1]
- [更正 2]

这显著改变了我的理解。[总结新的理解]
```

**如果用户确认假设：**

```
假设已验证。
```

继续执行 offer_next。
</step>

<step name="offer_next">
展示后续步骤：

```
下一步做什么？
1. 讨论上下文 (/gsd:discuss-phase ${PHASE}) - 让我向你提问以构建全面的上下文
2. 规划此阶段 (/gsd:plan-phase ${PHASE}) - 创建详细的执行计划
3. 重新检查假设 - 我将根据你的更正再次分析
4. 暂时结束
```

等待用户选择。

如果选择 "讨论上下文"：指出 CONTEXT.md 将包含此处讨论的所有更正
如果选择 "规划此阶段"：在已知假设已被理解的情况下继续
如果选择 "重新检查"：带着更新后的理解返回 analyze_phase
</step>

</process>

<success_criteria>
- 阶段编号已针对路线图验证
- 在五个领域揭示了假设：技术方案、实施顺序、范围、风险、依赖关系
- 在适当的地方标记了置信度
- 展示了 "你觉得怎么样？" 提示
- 确认了用户反馈
- 提供了清晰的后续步骤
</success_criteria>
