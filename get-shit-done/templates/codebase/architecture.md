# 架构模板

此模板用于 `.planning/codebase/ARCHITECTURE.md` —— 记录概念性的代码组织方式。

**目的：** 在概念层面记录代码的组织方式。作为 `STRUCTURE.md`（显示物理文件位置）的补充。

<good_examples>
```markdown
# 架构

**分析日期：** 2025-01-20

## 模式概览

**整体模式：** 带有插件系统的 CLI 应用程序

**关键特征：**
- 带有子命令的单个可执行文件
- 基于插件的可扩展性
- 基于文件的状态（无数据库）
- 同步执行模型

## 分层设计

**命令层 (Command Layer)：**
- 目的：解析用户输入并路由到相应的处理器
- 包含内容：命令定义、参数解析、帮助文本
- 位置：`src/commands/*.ts`
- 依赖于：业务逻辑的服务层
- 被谁使用：CLI 入口点 (`src/index.ts`)

**服务层 (Service Layer)：**
- 目的：核心业务逻辑
- 包含内容：FileService, TemplateService, InstallService
- 位置：`src/services/*.ts`
- 依赖于：文件系统工具、外部工具
- 被谁使用：命令处理器

**工具层 (Utility Layer)：**
- 目的：共享的辅助函数和抽象
- 包含内容：文件 I/O 封装、路径解析、字符串格式化
- 位置：`src/utils/*.ts`
- 依赖于：仅 Node.js 内置模块
- 被谁使用：服务层

## 数据流

**CLI 命令执行：**

1. 用户运行：`gsd new-project`
2. Commander 解析参数和标志
3. 调用命令处理器 (`src/commands/new-project.ts`)
4. 处理器调用服务方法 (`src/services/project.ts` → `create()`)
5. 服务读取模板、处理文件并写入输出
6. 结果记录到控制台
7. 进程以状态码退出

**状态管理：**
- 基于文件：所有状态都存储在 `.planning/` 目录中
- 无持久化内存状态
- 每次命令执行都是独立的

## 关键抽象

**服务 (Service)：**
- 目的：封装某个领域的业务逻辑
- 示例：`src/services/file.ts`, `src/services/template.ts`, `src/services/project.ts`
- 模式：类似单例（作为模块导入，不进行实例化）

**命令 (Command)：**
- 目的：CLI 命令定义
- 示例：`src/commands/new-project.ts`, `src/commands/plan-phase.ts`
- 模式：Commander.js 命令注册

**模板 (Template)：**
- 目的：可重用的文档结构
- 示例：PROJECT.md, PLAN.md 模板
- 模式：带有替换变量的 Markdown 文件

## 入口点

**CLI 入口：**
- 位置：`src/index.ts`
- 触发器：用户运行 `gsd <command>`
- 职责：注册命令、解析参数、显示帮助

**命令：**
- 位置：`src/commands/*.ts`
- 触发器：从 CLI 匹配的命令
- 职责：验证输入、调用服务、格式化输出

## 错误处理

**策略：** 抛出异常，在命令层捕获，记录日志并退出

**模式：**
- 服务抛出带有描述性信息的 Error
- 命令处理器捕获错误，将错误记录到 stderr，执行 exit(1)
- 在执行前显示验证错误（快速失败）

## 横切关注点

**日志记录：**
- 使用 Console.log 进行普通输出
- 使用 Console.error 记录错误
- 使用 Chalk 进行彩色输出

**验证：**
- 使用 Zod 模式解析配置文件
- 在命令处理器中进行手动验证
- 针对无效输入快速失败

**文件操作：**
- 对 fs-extra 封装的 FileService 抽象
- 在操作前验证所有路径
- 原子写入（临时文件 + 重命名）

---

*架构分析：2025-01-20*
*当主要模式发生变化时请更新*
```

<guidelines>
**ARCHITECTURE.md 包含的内容：**
- 整体架构模式（单体、微服务、分层等）
- 概念层级及其关系
- 数据流 / 请求生命周期
- 关键抽象和模式
- 入口点
- 错误处理策略
- 横切关注点（日志、认证、验证）

**不包含的内容：**
- 详尽的文件列表（那是 `STRUCTURE.md` 的职责）
- 技术选择（那是 `STACK.md` 的职责）
- 逐行代码讲解（交由代码阅读处理）
- 特定功能的实现细节

**欢迎包含文件路径：**
将文件路径作为抽象的具体示例包含在内。使用反引号格式：`src/services/user.ts`。这使得架构文档在 Claude 进行规划时具有可操作性。

**填写此模板时：**
- 阅读主要入口点（index, server, main）
- 通过阅读导入/依赖项来识别层级
- 追踪典型的请求/命令执行过程
- 注意重复出现的模式（服务、控制器、仓库）
- 保持描述是概念性的，而非机械性的

**对阶段规划有用的情况：**
- 添加新功能（它属于哪个层级？）
- 重构（理解当前模式）
- 确定代码添加位置（哪个层级处理 X？）
- 理解组件之间的依赖关系
</guidelines>
