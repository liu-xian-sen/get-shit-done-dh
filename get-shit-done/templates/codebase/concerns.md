# 代码库关注点模板

此模板用于 `.planning/codebase/CONCERNS.md` —— 记录已知问题和需要谨慎对待的领域。

**目的：** 提供关于代码库的可操作性警告。专注于“在进行更改时需要注意什么”。

<good_examples>
```markdown
# 代码库关注点

**分析日期：** 2025-01-20

## 技术债

**React 组件中的数据库查询：**
- 问题：在 15 个以上的页面组件中直接进行 Supabase 查询，而非使用 Server Actions
- 文件：`app/dashboard/page.tsx`, `app/profile/page.tsx`, `app/courses/[id]/page.tsx`, `app/settings/page.tsx` (以及 `app/` 下的另外 11 个文件)
- 原因：MVP 阶段的快速原型开发
- 影响：无法正确实施 RLS，向客户端暴露了数据库结构
- 修复方案：将所有查询移动到 `app/actions/` 中的 Server Actions，并添加正确的 RLS 策略

**手动的 Webhook 签名验证：**
- 问题：在 3 个不同的端点中复制粘贴了 Stripe Webhook 验证代码
- 文件：`app/api/webhooks/stripe/route.ts`, `app/api/webhooks/checkout/route.ts`, `app/api/webhooks/subscription/route.ts`
- 原因：每个 Webhook 都是临时添加的，缺乏抽象
- 影响：新的 Webhook 容易漏掉验证（安全风险）
- 修复方案：创建共享的 `lib/stripe/validate-webhook.ts` 中间件

## 已知 Bug

**订阅更新中的竞态条件：**
- 现象：用户在支付成功后 5-10 秒内仍显示为“免费”级别
- 触发条件：在 Stripe 结账重定向后、Webhook 处理前快速导航
- 文件：`app/checkout/success/page.tsx` (重定向处理器), `app/api/webhooks/stripe/route.ts` (Webhook)
- 绕过方案：Stripe Webhook 最终会更新状态（自我修复）
- 根本原因：Webhook 处理速度慢于用户导航，且无乐观 UI 更新
- 修复：在重定向后的 `app/checkout/success/page.tsx` 中添加轮询

**注销后会话状态不一致：**
- 现象：用户注销后重定向到 /dashboard 而非 /login
- 触发条件：通过移动端导航栏中的按钮注销（桌面端正常）
- 文件：`components/MobileNav.tsx` (第 45 行左右，注销处理器)
- 绕过方案：手动导航到 URL /login 可行
- 根本原因：移动端导航组件未等待 `supabase.auth.signOut()` 执行完成
- 修复：在 `components/MobileNav.tsx` 的注销处理器中添加 await

## 安全考量

**仅在客户端进行的管理员角色检查：**
- 风险：管理员仪表板页面仅通过 Supabase 客户端检查 isAdmin，无服务器端验证
- 文件：`app/admin/page.tsx`, `app/admin/users/page.tsx`, `components/AdminGuard.tsx`
- 当前缓解措施：无（仅依赖于 UI 隐藏）
- 建议：在 `middleware.ts` 中为管理员路由添加中间件，在服务器端验证角色

**未经验证的文件上传：**
- 风险：用户可以向 avatar 存储桶上传任何文件类型（无大小/类型验证）
- 文件：`components/AvatarUpload.tsx` (上传处理器)
- 当前缓解措施：Supabase 存储桶限制为 2MB（在仪表板中配置）
- 建议：在 `lib/storage/validate.ts` 中添加文件类型验证（仅限 image/*）

## 性能瓶颈

**/api/courses 端点：**
- 问题：获取所有课程时嵌套了课时和作者
- 文件：`app/api/courses/route.ts`
- 测量数据：包含 50+ 课程时，p95 响应时间为 1.2s
- 原因：N+1 查询模式（每个课程单独查询课时）
- 改进路径：在 `lib/db/courses.ts` 中使用 Prisma 的 include 进行预加载课时，并添加 Redis 缓存

**仪表板初始加载：**
- 问题：挂载时有 5 个串行的 API 调用瀑布流
- 文件：`app/dashboard/page.tsx`
- 测量数据：在慢速 3G 网络下需 3.5s 才能进行交互
- 原因：每个组件独立获取自己的数据
- 改进路径：转换为使用单个并行获取的 Server Component

## 脆弱领域

**身份验证中间件链：**
- 文件：`middleware.ts`
- 脆弱原因：4 个不同的中间件函数按特定顺序运行 (auth -> role -> subscription -> logging)
- 常见故障：中间件顺序更改会破坏一切，难以调试
- 安全修改方式：在更改顺序前添加测试，在注释中记录依赖关系
- 测试覆盖率：中间件链无集成测试（仅有单元测试）

**Stripe Webhook 事件处理：**
- 文件：`app/api/webhooks/stripe/route.ts`
- 脆弱原因：包含 12 种事件类型的巨型 switch 语句，共享事务逻辑
- 常见故障：添加新事件类型时未处理，出错时数据库部分更新
- 安全修改方式：将每个事件处理器提取到 `lib/stripe/handlers/*.ts`
- 测试覆盖率：12 种事件类型中仅 3 种有测试

## 扩展限制

**Supabase 免费层级：**
- 当前容量：500MB 数据库, 1GB 文件存储, 2GB 带宽/月
- 极限：预计达到约 5000 用户时触及极限
- 达到极限时的现象：429 频率限制错误，数据库写入失败
- 扩展路径：升级到 Pro ($25/mo)，扩展至 8GB 数据库, 100GB 存储

**服务器端渲染阻塞：**
- 当前容量：性能下降前约 50 个并发用户
- 极限：Vercel Hobby 计划 (10s 函数超时, 100GB-hrs/mo)
- 达到极限时的现象：课程页面出现 504 网关超时
- 扩展路径：升级到 Vercel Pro ($20/mo)，添加边缘缓存

## 有风险的依赖项

**react-hot-toast：**
- 风险：缺乏维护（最后一次更新在 18 个月前），React 19 兼容性未知
- 影响：Toast 通知失效，无优雅降级
- 迁移计划：切换到 sonner（活跃维护中，API 类似）

## 缺失的关键功能

**支付失败处理：**
- 问题：订阅支付失败时无重试机制或用户通知
- 当前绕过方案：用户手动重新输入支付信息（如果他们注意到了）
- 阻塞了：无法挽留信用卡过期的用户，无催款流程
- 实现复杂度：中（Stripe Webhooks + 邮件流 + UI）

**课程进度跟踪：**
- 问题：没有记录哪些课时已完成的持久化状态
- 当前绕过方案：用户手动跟踪进度
- 阻塞了：无法显示完成百分比，无法推荐下一课时
- 实现复杂度：低（添加 completed_lessons 关联表）

## 测试覆盖率缺口

**端到端支付流程：**
- 哪些内容未测试：完整的 Stripe checkout -> webhook -> 订阅激活流程
- 风险：支付处理可能在未察觉的情况下损坏（已发生过两次）
- 优先级：高
- 测试难度：需要 Stripe 测试 Fixtures 和 Webhook 模拟设置

**错误边界行为：**
- 哪些内容未测试：组件抛出错误时应用的行为
- 风险：用户看到白屏，无错误报告
- 优先级：中
- 测试难度：需要在测试环境中故意触发错误

---

*关注点审计：2025-01-20*
*当问题修复或发现新问题时请更新*
```

<guidelines>
**CONCERNS.md 包含的内容：**
- 具有明确影响和修复方案的技术债
- 带有重现步骤的已知 Bug
- 安全缺口和缓解建议
- 带有测量数据的性能瓶颈
- 容易损坏的脆弱代码
- 带有具体数值的扩展限制
- 需要关注的依赖项
- 阻塞工作流的缺失功能
- 测试覆盖率缺口

**不包含的内容：**
- 无证据的观点（"代码很乱"）
- 无解决方案的抱怨（"auth 很烂"）
- 未来的功能创意（那是产品规划的内容）
- 普通的 TODO（那些应存在于代码注释中）
- 运行良好的架构决策
- 轻微的代码风格问题

**填写此模板时：**
- **务必包含文件路径** —— 没有位置的关注点是不具有可操作性的。使用反引号：`src/file.ts`
- 测量数据要具体（"500ms p95" 而非 "慢"）
- 为 Bug 包含重现步骤
- 建议修复方案，而不仅仅是提出问题
- 专注于可操作的项目
- 按风险/影响排列优先级
- 当问题解决时进行更新
- 发现新关注点时及时添加

**语气准则：**
- 专业而非情绪化（"N+1 查询模式" 而非 "糟糕的查询"）
- 以解决方案为导向（"修复：添加索引" 而非 "需要修复"）
- 关注风险（"可能泄露用户数据" 而非 "安全很差"）
- 客观事实（"3.5s 加载时间" 而非 "真的很慢"）

**对阶段规划有用的情况：**
- 决定下一步的工作内容
- 估算更改的风险
- 了解哪里需要格外小心
- 排列改进工作的优先级
- 引导新的 Claude 上下文
- 规划重构工作

**如何填充内容：**
Explore 代理在代码库映射期间会检测这些内容。欢迎人工发现问题后进行手动添加。这是动态更新的文档，而非投诉列表。
</guidelines>
