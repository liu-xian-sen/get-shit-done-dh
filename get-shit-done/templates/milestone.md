# 里程碑条目模板

完成里程碑时，将此条目添加到 `.planning/MILESTONES.md`：

```markdown
## v[X.Y] [Name] (交付日期: YYYY-MM-DD)

**交付内容：** [用一句话描述已交付的内容]

**已完成阶段：** [X-Y] (总计 [Z] 个计划)

**关键成就：**
- [重大成就 1]
- [重大成就 2]
- [重大成就 3]
- [重大成就 4]

**统计数据：**
- [X] 个文件已创建/修改
- [Y] 行代码 (主要语言)
- [Z] 个阶段，[N] 个计划，[M] 个任务
- 从开始到交付 (或里程碑之间) 共 [D] 天

**Git 范围：** `feat(XX-XX)` → `feat(YY-YY)`

**下一步计划：** [简要描述下一个里程碑的目标，或“项目已完成”]

---
```

<structure>
如果 MILESTONES.md 不存在，请使用以下标题创建：

```markdown
# 项目里程碑：[项目名称]

[条目按时间倒序排列 - 最新的在前]
```
</structure>

<guidelines>
**何时创建里程碑：**
- 初始 v1.0 MVP 已交付
- 主版本发布 (v2.0, v3.0)
- 重大功能里程碑 (v1.1, v1.2)
- 在归档计划之前 (记录已交付的内容)

**不要为以下情况创建里程碑：**
- 单个阶段完成 (正常工作流)
- 进行中的工作 (等待交付后)
- 不构成版本的细微错误修复

**要包含的统计数据：**
- 统计修改的文件：`git diff --stat feat(XX-XX)..feat(YY-YY) | tail -1`
- 统计代码行数 (LOC)：`find . -name "*.swift" -o -name "*.ts" | xargs wc -l` (或相关的扩展名)
- 来自 ROADMAP 的阶段/计划/任务计数
- 从第一个阶段提交到最后一个阶段提交的时间线

**Git 范围格式：**
- 里程碑的第一条提交 → 里程碑的最后一条提交
- 示例：对于第 1-4 阶段，使用 `feat(01-01)` → `feat(04-01)`
</guidelines>

<example>
```markdown
# 项目里程碑：WeatherBar

## v1.1 安全与打磨 (交付日期: 2025-12-10)

**交付内容：** 通过 Keychain 集成和全面的错误处理实现了安全加固

**已完成阶段：** 5-6 (总计 3 个计划)

**关键成就：**
- 将 API 密钥存储从明文迁移到 macOS Keychain
- 实现了针对网络故障的全面错误处理
- 添加了 Sentry 崩溃报告集成
- 修复了自动刷新定时器中的内存泄漏

**统计数据：**
- 23 个文件已修改
- 添加了 650 行 Swift 代码
- 2 个阶段，3 个计划，12 个任务
- 从 v1.0 到 v1.1 共 8 天

**Git 范围：** `feat(05-01)` → `feat(06-02)`

**下一步计划：** v2.0 SwiftUI 重新设计并支持小组件

---

## v1.0 MVP (交付日期: 2025-11-25)

**交付内容：** 具有当前天气状况和 3 天预报的菜单栏天气应用

**已完成阶段：** 1-4 (总计 7 个计划)

**关键成就：**
- 带有弹出式 UI 的菜单栏应用 (AppKit)
- 带有自动刷新的 OpenWeather API 集成
- 带有状况图标的当前天气显示
- 带有最高/最低气温的 3 天预报列表
- 已完成代码签名和公证以供分发

**统计数据：**
- 47 个文件已创建
- 2,450 行 Swift 代码
- 4 个阶段，7 个计划，28 个任务
- 从开始到交付共 12 天

**Git 范围：** `feat(01-01)` → `feat(04-01)`

**下一步计划：** 为 v1.1 进行安全审计和加固
```
</example>
