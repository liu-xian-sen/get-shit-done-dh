# PROJECT.md 模板

用于 `.planning/PROJECT.md` 的模板 —— 动态的项目上下文文档。

<template>

```markdown
# [项目名称]

## 这是什么
[当前准确的描述 —— 2-3 句话。此产品的作用是什么，它是为谁服务的？
使用用户的语言和表述。每当实际情况偏离此描述时，请进行更新。]

## 核心价值
[最重要的一件事。如果其他一切都失败了，这件事也必须成功。
在出现权衡时驱动优先级排序的一句话。]

## 要求 (Requirements)

### 已验证 (Validated)
<!-- 已交付并确认有价值。 -->

(暂无 —— 交付后进行验证)

### 活跃 (Active)
<!-- 当前范围。正朝着这些目标构建。 -->

- [ ] [要求 1]
- [ ] [要求 2]
- [ ] [要求 3]

### 超出范围 (Out of Scope)
<!-- 明确的边界。包括理由以防止重新添加。 -->

- [排除项 1] — [原因]
- [排除项 2] — [原因]

## 上下文 (Context)
[为实现提供参考的背景信息：
- 技术环境或生态系统
- 相关的先前工作或经验
- 用户研究或反馈主题
- 需要解决的已知问题]

## 约束 (Constraints)
- **[类型]**: [内容] — [原因]
- **[类型]**: [内容] — [原因]

常见类型：技术栈、时间线、预算、依赖关系、兼容性、性能、安全

## 关键决策 (Key Decisions)
<!-- 约束未来工作的决策。在整个项目生命周期中添加。 -->

| 决策 | 理由 | 结果 |
|----------|-----------|---------|
| [选择] | [为什么] | [✓ 良好 / ⚠️ 重新审视 / — 待定] |

---
*最后更新：[日期] 在 [触发因素] 之后*
```

</template>

<guidelines>

**这是什么：**
- 对产品的当前准确描述
- 2-3 句话说明它的作用以及为谁服务
- 使用用户的词汇和表述
- 当产品演进超出此描述时进行更新

**核心价值：**
- 唯一最重要的一件事
- 其他一切都可以失败，唯独这件事不行
- 在出现权衡时驱动优先级排序
- 很少改变；如果改变，则意味着重大的转型 (pivot)

**要求 —— 已验证：**
- 已交付并证明有价值的要求
- 格式：`- ✓ [要求] — [版本/阶段]`
- 这些要求已锁定 —— 更改它们需要进行明确讨论

**要求 —— 活跃：**
- 正在构建的当前范围
- 在交付并验证之前，这些都是假设
- 交付后移至“已验证”，如果失效则移至“超出范围”

**要求 —— 超出范围：**
- 我们不构建的内容的明确边界
- 始终包含理由（防止以后重新添加）
- 包括：已考虑并拒绝的、延后到未来的、明确排除的

**上下文：**
- 为实现决策提供参考的背景
- 技术环境、先前工作、用户反馈
- 需要解决的已知问题或技术债
- 随着新背景的出现而更新

**约束：**
- 对实现选择的硬性限制
- 技术栈、时间线、预算、兼容性、依赖关系
- 包含“为什么” —— 没有理由的约束会受到质疑

**关键决策：**
- 影响未来工作的重大选择
- 在整个项目过程中随做随记
- 在已知结果时跟踪结果：
  - ✓ 良好 —— 决策证明是正确的
  - ⚠️ 重新审视 —— 决策可能需要重新考虑
  - — 待定 —— 现在评估还为时过早

**最后更新：**
- 始终注明文档更新的时间和原因
- 格式：`Phase 2 之后` 或 `v1.0 里程碑之后`
- 触发对内容是否仍然准确的审查

</guidelines>

<evolution>

PROJECT.md 在整个项目生命周期中不断演进。
这些规则嵌入在生成的 PROJECT.md 中（## 演进部分），
并由 workflows/transition.md 和 workflows/complete-milestone.md 实现。

**在每个阶段转换之后：**
1. 要求失效了？ → 移至“超出范围”并注明原因
2. 要求验证了？ → 移至“已验证”并注明阶段引用
3. 出现了新要求？ → 添加到“活跃”
4. 有决策需要记录？ → 添加到“关键决策”
5. “这是什么”仍然准确吗？ → 如果发生偏离请更新

**在每个里程碑之后：**
1. 对所有章节进行全面审查
2. 核心价值检查 —— 仍然是正确的优先级吗？
3. 审计“超出范围” —— 理由是否仍然有效？
4. 使用当前状态（用户、反馈、指标）更新“上下文”

</evolution>

<brownfield>

对于现有代码库：

1. 首先通过 `/gsd:map-codebase` **映射代码库**

2. 从现有代码中**推断“已验证”的要求**：
   - 代码库实际上做了什么？
   - 建立了哪些模式？
   - 哪些内容明显有效且被依赖？

3. 从用户那里**收集“活跃”要求**：
   - 呈现推断出的当前状态
   - 询问他们接下来想构建什么

4. **初始化：**
   - 已验证 = 从现有代码中推断
   - 活跃 = 用户对此项工作的目标
   - 超出范围 = 用户指定的边界
   - 上下文 = 包括当前代码库状态

</brownfield>

<state_reference>

STATE.md 引用 PROJECT.md：

```markdown
## 项目引用 (Project Reference)

参见：.planning/PROJECT.md (更新日期 [日期])

**核心价值：** [核心价值章节中的简短说明]
**当前焦点：** [当前阶段名称]
```

这确保了 Claude 读取当前的 PROJECT.md 上下文。

</state_reference>
