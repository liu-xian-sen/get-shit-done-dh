# PROJECT.md 模板

`.planning/PROJECT.md` 的模板 —— 动态的项目上下文文档。

<template>

```markdown
# [项目名称]

## 这是什么

[当前准确的描述 —— 2-3 句话。这个产品是做什么的，受众是谁？
使用用户的语言和表述。每当实际情况偏离此描述时请更新。]

## 核心价值

[最重要的一件事。如果其他都失败了，这件事必须成功。
在出现权衡时，驱动优先级的一句话。]

## 需求

### 已验证

<!-- 已交付并确认有价值。 -->

(暂无 —— 交付以验证)

### 活跃

<!-- 当前范围。正朝着这些目标构建。 -->

- [ ] [需求 1]
- [ ] [需求 2]
- [ ] [需求 3]

### 超出范围

<!-- 明确的边界。包含理由以防止重新添加。 -->

- [排除项 1] —— [原因]
- [排除项 2] —— [原因]

## 上下文

[为实现提供参考的背景信息：
- 技术环境或生态系统
- 相关的前期工作或经验
- 用户研究或反馈主题
- 需要解决的已知问题]

## 约束条件

- **[类型]**：[内容] —— [原因]
- **[类型]**：[内容] —— [原因]

常见类型：技术栈、时间线、预算、依赖项、兼容性、性能、安全

## 关键决策

<!-- 约束未来工作的决策。在整个项目生命周期中添加。 -->

| 决策 | 理由 | 结果 |
|----------|-----------|---------|
| [选择] | [原因] | [✓ 良好 / ⚠️ 重新审视 / — 待定] |

---
*最后更新：[日期] (触发因素：[trigger])*
```

</template>

<guidelines>

**这是什么：**
- 产品的当前准确描述
- 2-3 句话捕捉它做什么以及为谁服务
- 使用用户的原话和表述方式
- 当产品演进超出此描述时进行更新

**核心价值：**
- 唯一最重要的事情
- 其他一切都可以失败，但这个不能
- 在出现权衡时驱动优先级
- 很少改变；如果改变了，则意味着重大的转型

**需求 —— 已验证：**
- 已交付并证明有价值的需求
- 格式：`- ✓ [需求] —— [版本/阶段]`
- 这些是锁定的 —— 更改它们需要明确的讨论

**需求 —— 活跃：**
- 当前正在构建的目标范围
- 在交付和验证之前，这些都是假设
- 交付后移至“已验证”，如果被证明无效则移至“超出范围”

**需求 —— 超出范围：**
- 我们不构建的内容的明确边界
- 始终包含理由 (防止以后重新添加)
- 包括：已考虑并拒绝、推迟到未来、明确排除的内容

**上下文：**
- 为实现决策提供参考的背景
- 技术环境、前期工作、用户反馈
- 需要解决的已知问题或技术债
- 随着新背景的出现而更新

**约束条件：**
- 对实现选择的硬性限制
- 技术栈、时间线、预算、兼容性、依赖项
- 包含“为什么” —— 没有理由的约束会受到质疑

**关键决策：**
- 影响未来工作的重大选择
- 在整个项目过程中随做随记
- 在已知时跟踪结果：
  - ✓ 良好 —— 决策证明是正确的
  - ⚠️ 重新审视 —— 决策可能需要重新考虑
  - — 待定 —— 现在评估还为时过早

**最后更新：**
- 始终记录文档更新的时间和原因
- 格式：`after Phase 2` 或 `after v1.0 milestone`
- 触发对内容是否仍然准确的审查

</guidelines>

<evolution>

PROJECT.md 在整个项目生命周期中演进。

**在每个阶段转换后：**
1. 需求失效了？ → 移至“超出范围”并注明原因
2. 需求通过验证了？ → 移至“已验证”并注明阶段引用
3. 出现了新需求？ → 添加到“活跃”
4. 有决策要记录？ → 添加到“关键决策”
5. “这是什么”仍然准确吗？ → 如果发生偏移请更新

**在每个里程碑之后：**
1. 对所有部分进行全面审查
2. 核心价值检查 —— 仍然是正确的优先级吗？
3. 审计“超出范围” —— 理由仍然有效吗？
4. 用当前状态 (用户、反馈、指标) 更新“上下文”

</evolution>

<brownfield>

对于现有代码库：

1. **先通过 `/gsd:map-codebase` 映射代码库**

2. **从现有代码中推断“已验证”的需求：**
   - 代码库实际上做了什么？
   - 建立了哪些模式？
   - 什么是明确运行且被依赖的？

3. **从用户处收集“活跃”需求：**
   - 呈现推断出的当前状态
   - 询问他们接下来想构建什么

4. **初始化：**
   - 已验证 = 从现有代码推断出的内容
   - 活跃 = 用户针对此项工作的目标
   - 超出范围 = 用户指定的边界
   - 上下文 = 包含当前代码库状态

</brownfield>

<state_reference>

STATE.md 引用 PROJECT.md：

```markdown
## Project Reference

See: .planning/PROJECT.md (updated [date])

**Core value:** [来自“核心价值”部分的一句话总结]
**Current focus:** [当前阶段名称]
```

这确保了 Claude 读取当前的 PROJECT.md 上下文。

</state_reference>
