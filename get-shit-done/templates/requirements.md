# 需求模板

`.planning/REQUIREMENTS.md` 的模板 —— 定义“完成”的可检查需求。

<template>

```markdown
# 需求：[项目名称]

**定义日期：** [date]
**核心价值：** [来自 PROJECT.md]

## v1 需求

初始发布的需求。每个需求都对应路线图中的阶段。

### 身份验证 (Authentication)

- [ ] **AUTH-01**: 用户可以使用邮箱和密码注册
- [ ] **AUTH-02**: 用户注册后收到验证邮件
- [ ] **AUTH-03**: 用户可以通过邮件链接重置密码
- [ ] **AUTH-04**: 用户会话在浏览器刷新后保持

### [类别 2]

- [ ] **[CAT]-01**: [需求描述]
- [ ] **[CAT]-02**: [需求描述]
- [ ] **[CAT]-03**: [需求描述]

### [类别 3]

- [ ] **[CAT]-01**: [需求描述]
- [ ] **[CAT]-02**: [需求描述]

## v2 需求

推迟到未来发布的需求。已记录但不在当前路线图中。

### [类别]

- **[CAT]-01**: [需求描述]
- **[CAT]-02**: [需求描述]

## 超出范围

明确排除的内容。记录在案以防止范围蔓延。

| 功能 | 原因 |
|---------|--------|
| [功能] | [排除原因] |
| [功能] | [排除原因] |

## 可追溯性 (Traceability)

哪些阶段涵盖了哪些需求。在创建路线图期间更新。

| 需求 ID | 对应阶段 | 状态 |
|-------------|-------|--------|
| AUTH-01 | Phase 1 | Pending |
| AUTH-02 | Phase 1 | Pending |
| AUTH-03 | Phase 1 | Pending |
| AUTH-04 | Phase 1 | Pending |
| [REQ-ID] | Phase [N] | Pending |

**覆盖情况：**
- v1 需求：共 [X] 项
- 已映射到阶段：[Y] 项

*最后更新：[date] (触发因素：[trigger])*
```

</template>

<guidelines>

**需求格式：**
- ID：`[类别]-[编号]` (例如 AUTH-01, CONTENT-02, SOCIAL-03)
- 描述：以用户为中心、可测试、原子化
- 复选框：仅用于 v1 需求 (v2 需求尚不可执行)

**类别：**
- 从研究文件 FEATURES.md 的类别中得出
- 与领域惯例保持一致
- 典型类别：身份验证、内容、社交、通知、审核、支付、管理

**v1 vs v2：**
- v1：承诺的范围，将出现在路线图阶段中
- v2：已确认但推迟，不在当前路线图中
- 将 v2 移至 v1 需要更新路线图

**超出范围：**
- 带有理由的明确排除项
- 防止以后出现“为什么没包含 X？”的疑问
- 研究中发现的反面功能 (Anti-features) 应连同警告一起放在这里

**可追溯性：**
- 初始为空，在创建路线图期间填充
- 每个需求至少映射到一个阶段
- 未映射的需求 = 路线图缺漏

**状态值：**
- Pending: 尚未开始
- In Progress: 阶段活跃中
- Complete: 需求已验证
- Blocked: 等待外部因素

</guidelines>

<evolution>

**在每个阶段完成后：**
1. 将涵盖的需求标记为 Complete
2. 更新可追溯性状态
3. 记录任何范围发生变化的需求

**在路线图更新后：**
1. 验证所有 v1 需求是否仍被映射
2. 如果范围扩大，添加新需求
3. 如果缩减范围，将需求移至 v2 或“超出范围”

**需求完成标准：**
- 当满足以下条件时，需求为“Complete”：
  - 功能已实现
  - 功能已验证 (测试通过，人工检查完成)
  - 功能已提交

</evolution>

<example>

```markdown
# 需求：CommunityApp

**定义日期：** 2025-01-14
**核心价值：** 用户可以与志趣相投的人分享和讨论内容

## v1 需求

### 身份验证 (Authentication)

- [ ] **AUTH-01**: 用户可以使用邮箱和密码注册
- [ ] **AUTH-02**: 用户注册后收到验证邮件
- [ ] **AUTH-03**: 用户可以通过邮件链接重置密码
- [ ] **AUTH-04**: 用户会话在浏览器刷新后保持

### 个人资料 (Profiles)

- [ ] **PROF-01**: 用户可以创建带有显示名称的个人资料
- [ ] **PROF-02**: 用户可以上传头像图片
- [ ] **PROF-03**: 用户可以编写简介 (最多 500 字)
- [ ] **PROF-04**: 用户可以查看其他用户的个人资料

### 内容 (Content)

- [ ] **CONT-01**: 用户可以创建文本帖子
- [ ] **CONT-02**: 用户可以在帖子中上传图片
- [ ] **CONT-03**: 用户可以编辑自己的帖子
- [ ] **CONT-04**: 用户可以删除自己的帖子
- [ ] **CONT-05**: 用户可以查看帖子信息流

### 社交 (Social)

- [ ] **SOCL-01**: 用户可以关注其他用户
- [ ] **SOCL-02**: 用户可以取消关注用户
- [ ] **SOCL-03**: 用户可以点赞帖子
- [ ] **SOCL-04**: 用户可以评论帖子
- [ ] **SOCL-05**: 用户可以查看动态流 (已关注用户的帖子)

## v2 需求

### 通知 (Notifications)

- **NOTF-01**: 用户收到应用内通知
- **NOTF-02**: 用户收到新关注者的邮件通知
- **NOTF-03**: 用户收到对自己帖子评论的邮件通知
- **NOTF-04**: 用户可以配置通知偏好

### 审核 (Moderation)

- **MODR-01**: 用户可以举报内容
- **MODR-02**: 用户可以屏蔽其他用户
- **MODR-03**: 管理员可以查看被举报的内容
- **MODR-04**: 管理员可以删除内容
- **MODR-05**: 管理员可以封禁用户

## 超出范围

| 功能 | 原因 |
|---------|--------|
| 实时聊天 | 复杂度高，非社区核心价值 |
| 视频帖子 | 存储/带宽成本高，推迟到 v2+ |
| OAuth 登录 | 邮箱/密码对 v1 来说已足够 |
| 移动端应用 | Web 优先，之后再做移动端 |

## 可追溯性

| 需求 ID | 对应阶段 | 状态 |
|-------------|-------|--------|
| AUTH-01 | Phase 1 | Pending |
| AUTH-02 | Phase 1 | Pending |
| AUTH-03 | Phase 1 | Pending |
| AUTH-04 | Phase 1 | Pending |
| PROF-01 | Phase 2 | Pending |
| PROF-02 | Phase 2 | Pending |
| PROF-03 | Phase 2 | Pending |
| PROF-04 | Phase 2 | Pending |
| CONT-01 | Phase 3 | Pending |
| CONT-02 | Phase 3 | Pending |
| CONT-03 | Phase 3 | Pending |
| CONT-04 | Phase 3 | Pending |
| CONT-05 | Phase 3 | Pending |
| SOCL-01 | Phase 4 | Pending |
| SOCL-02 | Phase 4 | Pending |
| SOCL-03 | Phase 4 | Pending |
| SOCL-04 | Phase 4 | Pending |
| SOCL-05 | Phase 4 | Pending |

**覆盖情况：**
- v1 需求：共 18 项
- 已映射到阶段：18 项
- 未映射：0 项 ✓

*需求定义日期：2025-01-14*
*最后更新：2025-01-14 (初始定义后)*
```
